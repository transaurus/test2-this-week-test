## ⚛️ React {#react}

[![Bun 1.0](/emails/issues/159/bun.jpg)](https://bun.sh/blog/bun-v1.0)

[**Bun 1.0**](https://bun.sh/blog/bun-v1.0)

本週絕對不容錯過：Bun 已正式發布穩定版 v1.0。雖然此消息不直接關聯 React，但未來勢必影響所有 React 開發者。

Bun 是基於 Webkit 並以 Zig 編寫的完整 JavaScript 和 TypeScript 工具鏈。它為當今眾多前端工具提供全新超高速替代方案：Node.js、npx、Babel、esbuild、swc、webpack、Jest、Vitest、npm、Yarn、pnpm... 基準測試結果極其驚人且真實可信。

它與現有工具的相容性以及對混合 ESM/CJS 的支援，使其能夠輕鬆作為直接替代方案採用。Bun 通常能無需任何修改直接運行您的程式碼（否則會被[🐦視為錯誤](https://twitter.com/jarredsumner/status/1700497869160268155)）。Next.js、Remix 和 Astro 等 React 框架已獲得支援。將其作為 npm/Yarn/pnpm 或 Jest/Vitest 的替代品可能是較低風險的良好切入點。

## 📱 React-Native {#react-native}

[![Static Hermes](/emails/issues/159/hermes.jpg)](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement)

[**Static Hermes**](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement)

Static Hermes 上週在 React-Native EU 大會上宣布。我沒有參加，因此我正試圖透過閱讀簡報來解釋我所理解的內容，因為影片尚未在 YouTube 上發布。

Static Hermes 是一個實驗性專案，可讓您選擇性地將部分 TypeScript（或 Flow）程式碼編譯為原生程式碼。與基於位元組碼解譯的 Hermes 常規模式相比，這可帶來顯著的效能提升（10-20 倍）。它還能夠直接從 TypeScript 程式碼輕鬆整合原生 API（零成本 FFI，比 JSI 包裝快 15-80 倍）。

要實現這一點，TypeScript 必須先成為一種「健全」（sound）的語言。Static Hermes 將修改 JavaScript 的語義，以確保宣告的類型與運行時類型相符。以下這張投影片很好地解釋了這個概念：

[![Static Hermes makes TypeScript sound](/emails/issues/159/hermes2.jpg)](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement?slide=14)

這個非常有趣的想法，也許有一天會在 React-Native 生態系統之外產生影響？
