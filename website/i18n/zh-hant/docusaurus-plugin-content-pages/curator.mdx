# This Week In React - 策展人職位說明

我最初創建 [This Week In React](https://thisweekinreact.com/) 作為策展型通訊，協助 React 開發者掌握生態系最新動態。

長期目標是成為真正由社群驅動的 React 與 React Native 開發者媒體（類似 CSS-Tricks）。我希望它不僅是每週發送連結清單的電子報。

為達成此目標，需要委派工作並招募協作夥伴協助產出週期內容。

這是**實際工作**：您將獲得報酬，需具備**可靠性**，並願意為單期電子報投入**14-18小時工作量**。

## 對 This Week In React 的益處

本次招募的主要目標：

- 建立可靠備援機制，避免我休假時出現斷更導致倦怠
- 透過每年 50+ 期內容塑造專業媒體形象
- 強化媒體的社群驅動屬性
- 透過輪替作者提升多元性
- 騰出時間深化媒體發展
- 確保通訊品質維持高水準

近五年我幾乎獨力運作，初期有趣的事可能成為長期負擔。我希望能自主決定發布時機而非被迫產出，最終讓專案能脫離個人獨立運作。

## 客座策展人的收益

您將獲得的主要收益：

- 每期通訊獲得**合理報酬**
- 提升**個人品牌**及所屬**企業能見度**（[客座策展人範例](https://thisweekinreact.com/newsletter/206)）
- 回饋社群，貢獻於實用專案

若有創新想法，我樂於討論其他收益方案，前提是達成雙贏。

截至目前，入職流程如下：

- 您自我介紹（或介紹貴公司員工），說明為何認為自己是合適人選

- 我們協作完成首期測試版通訊，讓您熟悉流程並掌握細節
- 您全權負責第二期測試版通訊。我會解答問題並提供回饋，但您需自主完成主要工作
- 我們檢視成果。雖無法完美，但只要我對發送的郵件不感慚愧，即表示運作正常且未來可持續優化
- 我們可規劃您後續負責的其他日期

## 編輯方針

**核心原則:**

- 針對**中高階 React 及 React Native 開發者**，但不限 React 相關連結
- **新鮮內容**。一個月前的資訊即屬過時，通常已涵蓋
- **捕捉弱訊號**以預見未來趨勢
- **事件驅動內容**，例如新版本發布、專案啟用或文件更新
- 成為生態系**脈搏監測器**，而非單純的 React npm 套件探索工具

**優先選擇:**

- 專注**單一概念深度解析**的內容
- 探討**React 核心技術**且具廣泛適用性的內容
- 呈現**新穎**、**創新**或**被低估**技術的內容
- 針對**利基主題**與**弱訊號**且呈現優良/具影響力的內容
- 具**卓越呈現效果**與**流暢閱讀體驗**的內容（例如互動式文章）

**我們要避免的內容：**

- 過濾人工智慧生成的內容（基於SEO考量）、清單式文章和其他低品質文章
- 過濾需付費訂閱、需登入驗證或閱讀體驗不佳的內容
- 過濾針對初階開發者的內容、基礎教學、逐步操作指南、單純複述官方文件而無附加價值的內容
- 過濾教學如何使用付費產品的內容（我們有贊助機制處理這類需求），除非內容不過度推銷且能傳授React核心有價值的知識
- 過濾大多數UI函式庫和樣板工具，數量實在太多。例外情況是當它們證明具高關注度、長期維護承諾或來自權威來源。我們不報導這類工具的「發佈消息」，但可能在其v10版本成為主流並證明社群價值時進行報導
- 過濾過於薄弱的弱訊號：若您無法理解內容、無法用簡單語言解釋其影響，或認為僅少數人受益，則可能為時過早或不值得收錄
- 盡可能避免X/Twitter串文/對話連結（需帳號登入）。優先選擇外部連結來源，個別推文若包含完整資訊、連結或影片演示則可接受

對於呈現效果極佳的內容可例外處理。例如解釋基礎React概念的互動式文章就非常合適：這是我們目標受眾（資深開發者）可能分享給初階同事的優質資源

## 內容

本通訊包含大量連結，因React生態系蓬勃發展。
我們不採取更嚴格過濾，而是優化資訊密集度下的閱讀體驗。
內容需保持一致性、易於掃讀，並盡可能「內嵌」資訊。
避免引誘點擊連結卻造成失望。
即使不點擊任何連結，閱讀通訊本身也應提供價值

### 章節

通訊包含5個主要章節：

- 導言：新聞簡要回顧
- React：適用所有React開發者及網頁開發者的內容
- React Native：React Native開發者相關內容
- 其他：更廣泛JS受眾內容，含TypeScript、CSS、TC39、HTML、工具鏈等
- 趣聞：1-3則迷因或趣味截圖，優先選擇與React（內部梗）及當週新聞相關

### 頭條

我們通常挑選：

- 1則React頭條
- 1則React Native頭條

每則頭條通常配有頂部橫幅圖。
優先選擇具視覺意義的圖像（如程式碼截圖），精美社群卡片亦可接受

有時某章節無明確頭條，有時則有2則。
偶爾會為「其他」章節挑選頭條

### 格式

多數連結遵循此格式：

- \<表情符號\> [連結標題](https://example.com)：可選主觀評論

若標題過長、屬點擊誘餌或大小寫異常，可調整格式使所有標題保持相對一致

版本發佈資訊需省略「v」和修訂號（例如3.1而非v3.1.9）。將主要更新亮點關鍵字直接包含在連結標題中，以短橫線`-`分隔。若無值得強調的更新（常見於修訂版），則不收錄。連結「內部」為「客觀」部分（更新亮點），「外部」為主觀評論（若有）。範例：

- 📦 [Astro 5.2 - Tailwind 4 支援、尾部斜線重新導向、停用 React streaming 的選項](https://astro.build/blog/astro-520/)：串流可能與 runtime CSS-in-JS 函式庫不相容。
- 📦 [Nextra 4.0 - 現採用 App Router](https://the-guild.dev/blog/nextra-4)
- 📦 [React Scan 0.1.2 - 重新渲染歷史紀錄、元件樹狀結構](https://github.com/aidenybai/react-scan/releases/tag/v0.1.2)

若為首次收錄的新專案，可省略版本號，並以客觀專案描述取代關鍵字（靈感來自 GitHub 儲存庫說明、登陸頁面、SEO 標籤）：

- 📦 [Polygen - 在應用程式中靜態執行 WebAssembly 模組，無需直譯](https://github.com/callstackincubator/polygen)：此 Callstack 預覽套件將 Wasm 轉換為 C 語言，透過 JSI 調用以實現卓越效能。

### 評論

需遵循的通用規則：

- 無需評論所有內容
- 特定表情符號的評論通常置於首位
- 與其發表錯誤評論，不如保持沉默
- 評論須簡明扼要
- 使用簡潔語言
- 避免添加無意義的冗詞
